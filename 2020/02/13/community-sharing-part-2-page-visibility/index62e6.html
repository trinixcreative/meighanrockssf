Community Sharing Part 2: Guest Page Visibility

What is a guest?

Every Community has a Guest User and Guest User Profile. Guest Users are users that can see your community before they login – an Unauthenticated user. Each time a community or portal is created, Salesforce auto-creates a guest user profile and a site guest user record. No matter what. Even if your community is authenticated, you have a guest user. The site guest user record is the only user record associated with the guest user profile. The Guest User works like other user records in Salesforce, so you can add it to assignment rules, queues, public groups, manual sharing, and permission sets. 

Unique Quirks of Guest Users

  • The guest user profile is specific to the community or portal. You will want to check the guest user profile for each of your communities or portals to ensure object/data security (we will review that later).
  • If the community is active, guest users can access a subset of community pages, such as login and error pages.  From there you can expand the pages visible by making pages or your community as a whole public.
  • Use the Guest User Access Report AppExchange package to understand if records are being shared with guest users (this is explained in the tips and tricks sections).

How do you access the guest profile? Well you can do this in 2 different ways:

  • From Builder go to the Gear on the Left Side, go to: General – Click the Guest Profile Name
  • From Setup go to Sites: click your Community Name – Select Public Access – You’re on your Guest User

Pages

How to Setup Communities to be Private or Public

If you allow public access, your community pages are accessible to the public, including unlicensed users. If don’t allow public access, members must log in to access the community.

  • Enable public access in a Lightning Community, open Community Builder.
    • In Classic: Setup go to Customize – All Communities click Builder by your community name
    • In Lightning: Setup go to Feature Settings – Communities – All Communities – click Builder next to your community’s name
    • From a community, click Community Builder in the profile menu.
  • Click the gear on the left side for the settings and stay on General.
  • Select Public can access the community.

You can also set page-level access on a page by page basis in your Experience Builder in Page Properties. You have 3 different level access settings:

  • Community Default Setting – Reflects your choice for public access under General Settings.
  • Public – Makes the page public regardless of the community’s default setting.
  • Requires Login – Makes the page private and requires members to log in, regardless of the community’s default setting.

Audience criteria-based page visibility in Community Builder

  1. When a member is trying to access a page, Salesforce first checks the community’s default setting. Is it public or does it require users to log in? If it requires Login and the user is a guest, they will redirect to the Login page, if they have access –>
  2. Then Salesforce looks at the page access.
  3. Lastly, Salesforce checks that the audience criteria-based visibility that you set in Page Variations.

Standard page visibility in Community Builder Settings

object page Settings to show data in Community Builder

Object pages don’t have the ability to set the page visibility, instead it is based on object access and page variations.  You can grant access to an object via the Guest User Profile and the Guest User will have access to the object pages. Then, assign the guest a different layout using variations.

It is also an option to modify the visibility on the component level, making specific components, such as Tiles, the list view component, and make them display for only some users.  To make a component on a public page visible to guest users, select Publicly available in the component’s properties.

Page Details & Config

  • Regardless of the settings, some pages are always public, and others are always private. Public pages include login-related pages (Login, Register, Forgot Password, Login Error, Check Password). The Messages page for direct messages is always private.
  • If public access is enabled in Community Builder at the page or community level, the Let guest users view asset files on public pages preference is enabled at Workspaces | Administration | Preferences.  This preference lets guest users view asset files shared with the community on publicly accessible pages. It remains enabled if the page has public access enabled.

Community Builder is a powerful tool for keeping your Lightning community or portal secure. Keep these best practices and considerations in mind when working with access levels in Lightning communities.

  • Don’t create object pages for objects not exposed to community users. Delete object pages that aren’t being used.
  • Use generic record pages when possible.
  • Make sure you remove chatter from the generic record page.
  • Update the record detail pages to use the tabs with details and related & feed on a content with sidebar layout vs full width
  • If you set up object pages for authenticated users, test the pages to ensure that guest users can’t see them.

Best Practices for Guest Users on Object Pages

  • Out of the box, when guest users create records, they are redirected to the homepage. To send to a different screen, create a flow and use a flow action for redirection. *Read ton before just redirecting* (Unofficial Flow Site Lightning Actions)
  • If you go custom and decide to send the Guest User to the record just created, remember that guest users loses access to the record they just created since it is reassigned immediately. Consider taking the user to a page other than the record detail, as if you share the record with the guest, there is only 1 guest on your community so you would be sharing with all users that are unauthenticated.
    • Salesforce recommends considering changing your implementation to redirect the guest user to a different part of your site after creating a record.
  • Guest users may have the option of associating a newly created record to a parent record via a lookup. An example is creating a contact and associating it to an account. Because guest users lose access after they create them, if the guest creates an Account with a related Contact, they would get an error saving the Contact because they lost access to the Account after creating it.  To avoid errors, you can:
    • Ensure that your org has a generic parent record shared with guest users via a sharing rule, (remember from above how that Guest User is just a normal user so they can be added to Sharing Rules) so new children are visible.
      • REMEMBER Your generic record shouldn’t have any private or sensitive data you don’t want shared out with the whole world
    • Make sure to remove unnecessary create permissions for guests.

Search

Search and your guest Users

The Guest User Profile controls what unauthenticated users can see in your community, be it public or private.  The Guest User Profile sees what your unauthenticated users see. To make an object’s records available for search indexing, you must provide at least read access to that object and its fields to the Guest User Profile.  Keep in mind this opens up the records shared with the Guest User has with the whole wide world.

SEO

  1. In Community Builder, make sure that the community is publicly available. Select Public can access the community in Settings | General, this also enables the SEO settings tab.
  2. On the SEO settings tab, configure which objects are public using the Guest User Profile.
  3. Click Guest User Profile.
  4. Click Edit, and in the Standard Object Permissions section, give at least read access to the object.
  5. Save your changes.
  6. In the Field-Level Security section, select View beside the object and ensure that the appropriate fields have at least read access.
  7. Save your changes.
  8. In Community Builder, create object pages for the object, if they don’t exist.
  9. Activate your community, if you haven’t yet. **NOTE: Always review which objects your Guest User Profile has access to before activating and publishing your community. This best practice controls your sitemap.xml file and helps keep your community safe & secure.**
  10. To make your changes live, publish your community.

It can take up to 24 hours for the list of available objects to appear, along with the number of indexed records.

Search Best Practices

When setting up your search pages and components for communities and portals, keep these best practices and considerations in mind to keep data secure from guest users.

Search Page

  • Review page access settings for the search page to ensure that you want the search page accessed by guest users.
  • To limit access to the search page, consider creating a search page variation with a guest audience.

Global Search Results Component

The Global Search Results component is the main component on the Search page in any community. The component allows admins to select the objects shown in search results.

  • Guest users only see results on objects they have access to.
  • Actual record access for the guest user isn’t limited to the objects the admin configures in the Global Search Results component. The guest user can have access to other objects based on org sharing configurations.
  • Always check the org’s sharing model (including org-wide defaults and sharing rules) to ensure that the guest user doesn’t have access to your org’s sensitive data.

Search Box Component

  • As with the Global Search Results component, the admin sets up which objects to show in searches. However, record access for the guest user isn’t limited to the objects the admin configures in the Global Search Box and Global Search for Peer-to-Peer Communities components. The guest user can have access to other objects based on org sharing configurations.
  • Always check the org’s sharing model (including org-wide defaults and sharing rules) to ensure the guest user doesn’t have access to your org’s sensitive data.

Well folks, that’s Community Sharing Part 2, our first section on guests and sharing/security.  Next up we’ll jump into the how does that Guest sharing on objects work, best practices, SEO and what settings should you be concerned with.  All on our exciting adventure through Community Sharing in Salesforce.
